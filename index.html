Dar de alta un cliente nuevo

Tiempo total: 40-60 min. Esta guía asume que este admin y los paneles ya están desplegados en Vercel (si estás leyendo esto acá, lo están). Lo de abajo es lo que se repite por cada cliente.

⚠️ Dos reglas de oro: (1) No abrir los archivos del paquete con TextEdit (Mac): rompe las comillas y el código falla. Usar VS Code o abrir con Chrome y copiar. (2) Credenciales SIEMPRE nuevas por cliente, nunca reutilizar.

Archivos necesarios (paquete nivel2): apps-script-backend.js, worker-encuesta-bundle.js, un seed de plantilla, gtm-loader.html.

Guardar la ficha completa donde Empuje guarda credenciales de clientes.

Paso 1 — Generar credenciales (2 min)
  1. ADMIN_TOKEN: 40+ caracteres aleatorios (llave maestra del Worker, interna Empuje).
  2. CLIENT_KEY: 32+ caracteres aleatorios (solo-lectura de métricas).
  3. Parámetro de preview: prefijo propio, ej. La Planchetta → lp_survey.

Pedírselas a Claude ("generame un admin token y una client key para [cliente]") o usar un generador de contraseñas. Anotarlas en la ficha YA.

Paso 2 — Google Sheet + Apps Script (10 min)
  1. Crear un Sheet: Encuesta de salida — [Cliente].
  2. Menú Extensiones → Apps Script → borrar el ejemplo → pegar TODO apps-script-backend.js.
  3. En la línea var CLIENT_KEY = '...' pegar la CLIENT_KEY del Paso 1. Guardar.
  4. Implementar → Nueva implementación → engranaje ⚙ → Aplicación web → Ejecutar como: Yo → Acceso: Cualquier usuario (crítico) → Implementar → autorizar permisos.
  5. Copiar la URL que termina en /exec. A la ficha.
  6. Verificar en incógnito: la URL /exec debe responder {"result":"ok","status":"vivo"}. Si pide login de Google → el acceso quedó en "Solo yo": corregir con Administrar implementaciones → lápiz → Nueva versión.
  7. Verificar métricas: /exec?action=metrics&key=LA_CLIENT_KEY → {"result":"ok","total":0,...}.

🔄 Ediciones futuras del script: siempre "Nueva VERSIÓN" sobre la implementación existente. Nunca "Nueva implementación" (cambia la URL y rompe todo).

Paso 3 — Cloudflare: Worker + KV (15 min)
  1. Cuenta: la del cliente si ya tiene (ej. por el quiz); si no, crear Gmail dedicado y cuenta Free en dash.cloudflare.com.
  2. KV: menú lateral → Storage & Databases → KV → Create a namespace → [cliente]-encuesta-CONFIG.
  3. Armar el seed (el paso con criterio): partir de un seed existente y editar: endpoint (la /exec del Paso 2 — olvido n°1), question.options (las objeciones DEL RUBRO de este cliente, definirlas con Claude), checkoutButtonSelectors y cartSelectors (inspeccionar el theme: click derecho sobre el botón de compra / contador del carrito → Inspeccionar → copiar la clase), previewParam, storeDomain, branding.
  4. Cargar el seed: entrar al namespace → pestaña KV Pairs → Add entry → Key: config:default (exacto) → Value: el JSON completo → Save.
  5. Worker: Compute (Workers) → Workers & Pages → Create → Create Worker → nombre [cliente]-encuesta → Deploy → Edit code → borrar todo → pegar TODO worker-encuesta-bundle.js (idéntico para todos) → Deploy.
  6. Binding: Settings → Bindings → + Add → KV namespace → Variable name: CONFIG (exacto, mayúsculas) → el namespace del cliente → Save.
  7. Secret: Settings → Variables and Secrets → + Add → Type: Secret → Name: ADMIN_TOKEN → Value: el token del Paso 1 → Save.
  8. Verificar: .../health responde ok; .../widget.js muestra JS que empieza con (function () { y adentro (Ctrl+F) aparece la /exec del cliente. La URL del Worker a la ficha.

Nota: /health dice "service":"topwhite-encuesta" en todos los Workers; es una etiqueta interna del bundle, no significa nada.

Paso 4 — GTM: el loader (5 min + permiso de publish)
  1. Abrir gtm-loader.html → reemplazar WORKER_URL por la URL del Worker, sin barra final.
  2. En el GTM del cliente: Etiquetas → Nueva → HTML personalizado → pegar → Activador: All Pages → nombrar [Cliente] - Encuesta de salida (loader) → Guardar.
  3. Si hay una versión vieja de la encuesta como tag gigante: pausarla o borrarla (si conviven, doble carga).
  4. Publicar el contenedor (único publish que el sistema pide en su vida). Sin publish, se puede probar igual con Vista previa de Tag Assistant.
Paso 5 — Alta en ESTE admin (1 min)
  1. Volver → + Agregar cliente → nombre, URL del Worker, ADMIN_TOKEN → Conectar y guardar.
  2. Si carga el formulario con la config del seed, el circuito completo funciona.

El alta queda en TU navegador. Cada compañero agrega el cliente en el suyo con los datos de la ficha.

Paso 6 — Test end-to-end (10 min — NO saltear el celular)
  1. Preview desktop: https://[tienda]/?[previewParam]=preview → mouse al borde superior → aparece → responder → fila en el Sheet con Origen = preview.
  2. Preview en CELULAR real (crítico): mismo link → botón atrás → aparece. Verificar que después el botón atrás siga navegando normal (si queda raro, avisar a Claude).
  3. Flujo real (sin parámetro): navegar tranquilo (+45s de sesión), carrito, "Iniciar compra", volver → a los ~2,5s aparece → fila con Origen = real.
  4. Negativo: sesión nueva sin iniciar checkout → no aparece nunca (correcto).
  5. Circuito admin: cambiar un texto acá → Guardar → 2 min → recargar el preview → se ve el cambio.
  6. Mandar el link de preview al cliente: "abrilo en tu celu y tocá el botón atrás para verla".
Pasos 7 y 8 (opcionales) — Panel de métricas y editor del cliente

7. Panel de métricas: abrir /cliente-index.html en incógnito → setup: nombre, /exec, CLIENT_KEY → Generar link → mandárselo por canal privado (lleva la clave adentro). No enviarlo hasta que haya un par de semanas de data.

8. Editor del cliente: en este admin → el cliente → tarjeta "Acceso del cliente (editor)" → definir usuario y clave → Guardar → copiar el link del editor y mandarlo junto con las credenciales. Revocable al instante desde la misma tarjeta.

Los 5 errores de este proceso
  1. Apps Script con acceso "Solo yo" (a vos te anda, a la tienda no). Verificar SIEMPRE en incógnito.
  2. No reemplazar el endpoint en el seed o WORKER_URL en el loader.
  3. El binding con otro nombre que no sea CONFIG exacto → error 1101.
  4. "Nueva implementación" en vez de "Nueva versión" al editar el Apps Script.
  5. Reutilizar tokens o claves entre clientes.
